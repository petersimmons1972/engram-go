Purpose

Three chunking improvements — borrowed conceptually from the Python library chonkie-inc/chonkie and re-implemented natively in Go — are intended to lift engram-go's LME-S and LME-M benchmark scores by improving how session content is chunked at ingest time. The plan covers: socialization of the approach with the AI fleet (Codex, Hermes, Grok); TDD-first Go implementation of each concept; and benchmark validation gating each phase's green-light.

All four LME-S question types are in scope: temporal-reasoning, single-session-preference, knowledge-update (KU-78), and multi-session. Both LME-S (500Q) and LME-M (500-session) are target benchmarks.

Problem

engram-go's ChunkText() at internal/engine/engine.go:~502 already accepts an overlapTokens parameter, but this value is hard-wired at the call site to 50 tokens — there is no CLI flag to tune it. No session-boundary flush exists, creating potential PII leakage between sessions when overlap is non-zero. Turn boundaries (user/assistant alternation within a session) are not respected as natural chunk split points. No semantic similarity check is applied at split time, meaning thematically coherent content can be split mid-thought.

The consequence: atom retrieval quality is sub-optimal, leading to a context window that does not reliably contain the most relevant turns for question-answering.

QA blocker corrections (6-persona review, prior session) — must be incorporated before any implementation begins:
  1. --max-block-chars 8000 targets the prompt-assembly layer, not the chunker. The new flag must be --block-overlap-chars N (separate namespace).
  2. --semantic-chunk boolean is ambiguous. Correct form: --semantic-chunk-threshold=<float> --semantic-chunk-model=<id>
  3. Parse-time guard required: if overlap >= max-block/2, exit nonzero with clear message. Server-enforced cap prevents runaway configuration.
  4. Overlap buffer must flush on session boundary. A new session ID must never inherit overlap tokens from a prior session (confidentiality, PII).

Solution

Native Go implementation of three concepts from Chonkie, applied sequentially with benchmark gates between phases. The Python library is not imported (Python/Go barrier, RCE surface, HuggingFace weight downloads); only the algorithms are adopted.

Phase Concept (Chonkie origin) Go target Gate
1 OverlapRefinery — sliding window token overlap between chunks Surface overlapTokens via --block-overlap-chars N; session-boundary flush KU-R2 clean run; ≥+1pp vs KU-B1 on paired items
2 Turn-boundary respect — never split user/assistant turns mid-turn ChunkText() split points snap to \nRole: boundaries ≥+1pp vs Phase 1 on multi-session question type
3 SemanticChunker — cosine similarity at split time --semantic-chunk-threshold=<float> --semantic-chunk-model=<id>; local embedding ≥+3pp delta vs Phase 2 baseline; else skip to Phase 4
4 Full LME-M validation Run complete 500Q on best config from Phases 1–3 >78.6% LME-M (beat camp-22)

Before implementation, the approach is socialized in parallel with three AI reviewers (Codex, Hermes, Grok) via the fleet-dispatch queue. Verdicts are synthesized, blockers addressed, and the final implementation brief is dispatched to Codex for Go coding.

Q2 — Session boundary detection: how does run.go identify a session boundary?

From types.go: Item.HaystackSessionIDs []string and Item.HaystackSessions [][]Turn — sessions are already pre-segmented in the input data structure. The boundary is simply the transition from one session slice to the next in the outer loop.

Assumption: The ingest loop in run.go iterates over item.HaystackSessions with an index; session boundary = index increment. This must be confirmed at implementation time by reading the actual run.go loop structure.

What would change this: If sessions are flattened before chunking (all turns concatenated into one string), a different boundary detection mechanism is needed — likely a sentinel delimiter like ---SESSION-BREAK---.
